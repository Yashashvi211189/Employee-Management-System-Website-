# Employee Management System Website

[![Django](https://img.shields.io/badge/Django-3.2.7-092E20?logo=django&logoColor=white)](https://www.djangoproject.com/)
[![Next.js](https://img.shields.io/badge/Next.js-14.2-black?logo=next.js)](https://nextjs.org/)
[![Python](https://img.shields.io/badge/Python-3.10%2B-3776AB?logo=python&logoColor=white)](https://www.python.org/)
[![TypeScript](https://img.shields.io/badge/TypeScript-5.x-3178C6?logo=typescript&logoColor=white)](https://www.typescriptlang.org/)

A full-stack employee management and company website platform with a Django REST backend, JWT authentication, employee operations, project tracking, content management, and a Next.js frontend.

## Key Features

- Employee login, onboarding, OTP verification, password changes, and profile access.
- Admin APIs for employees, documents, leave requests, overtime requests, tickets, ticket comments, ticket notifications, and leave balances.
- Project and private-project management with memberships, plans, assignments, and daily updates.
- Public website modules for services, GIS services, products, gallery, team members, blogs, testimonials, and contact pages.
- JWT authentication through Django REST Framework Simple JWT, with cookie-aware authentication support.
- OpenAPI schema, Swagger UI, and Redoc documentation generated by `drf-spectacular`.
- Optional S3-compatible storage and Twilio/email integrations through environment variables.
- Next.js App Router frontend with Tailwind CSS, Radix UI components, admin pages, employee dashboard pages, and static media assets.

## Architecture Overview

```text
Browser / Next.js frontend
        |
        | NEXT_PUBLIC_API_URL
        v
Django REST API + Django admin
        |
        +-- PostgreSQL database
        +-- Local media or S3-compatible object storage
        +-- SMTP email provider
        +-- Optional Twilio credentials
```

The repository is organized as a monorepo:

- `backend/` contains the Django project and REST APIs.
- `frontend/` contains the Next.js application.
- Root-level configuration keeps local generated output, credentials, virtual environments, and dependency folders out of Git.

## Tech Stack

| Layer | Technologies |
| --- | --- |
| Backend | Python, Django 3.2, Django REST Framework, Simple JWT, Channels, drf-spectacular |
| Database | PostgreSQL by default, SQLite available through `myproject.settings_local` |
| Storage | Local filesystem by default, optional S3-compatible storage via `django-storages` |
| Frontend | Next.js 14, React 18, TypeScript, Tailwind CSS, Radix UI, Lucide React |
| Deployment | Gunicorn/systemd/nginx examples for backend; Vercel and Netlify config for frontend |

## Prerequisites

- Python 3.10 or newer
- Node.js 18 or newer
- PostgreSQL for the default backend configuration
- Redis if you enable Django Channels-backed realtime features
- Git

## Backend Setup

```bash
cd backend
python -m venv .venv

# Windows PowerShell
.venv\Scripts\Activate.ps1

# macOS/Linux
# source .venv/bin/activate

pip install -r requirements.txt
copy .env.example .env
python manage.py migrate
python manage.py createsuperuser
python manage.py runserver 127.0.0.1:8000
```

For SQLite-based local development:

```bash
cd backend
set DJANGO_SETTINGS_MODULE=myproject.settings_local
python manage.py migrate
python manage.py runserver 127.0.0.1:8000
```

On macOS/Linux use `export DJANGO_SETTINGS_MODULE=myproject.settings_local`.

## Frontend Setup

```bash
cd frontend
npm install
npm run dev
```

The frontend runs on `http://localhost:3000` by default. The backend defaults to `http://127.0.0.1:8000`.

## Environment Variables

### Backend

Create `backend/.env` from `backend/.env.example`.

| Variable | Purpose |
| --- | --- |
| `DJANGO_SECRET_KEY` | Django signing secret. Required for production. |
| `DJANGO_ALLOWED_HOSTS` | Comma-separated host allowlist. |
| `BASE_URL` | Backend base URL used by API responses and integrations. |
| `FRONTEND_URL` | Frontend URL used by Django admin and links. |
| `DB_ENGINE` | Django database engine. Defaults to PostgreSQL. |
| `DB_NAME`, `DB_USER`, `DB_PASSWORD`, `DB_HOST`, `DB_PORT` | Database connection settings. |
| `EMAIL_HOST`, `EMAIL_PORT`, `EMAIL_HOST_USER`, `EMAIL_HOST_PASSWORD` | SMTP settings for email delivery. |
| `DEFAULT_FROM_EMAIL`, `CONTACT_EMAIL`, `EMAIL_USE_TLS` | Email sender/contact behavior. |
| `TWILIO_ACCOUNT_SID`, `TWILIO_AUTH_TOKEN` | Optional SMS/OTP integration credentials. |
| `ALWAYS_UPLOAD_FILES_TO_AWS` | Enables S3-compatible storage when set to `true`. |
| `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `AWS_STORAGE_BUCKET_NAME`, `AWS_S3_ENDPOINT_URL` | S3-compatible storage settings. |

### Frontend

| Variable | Purpose |
| --- | --- |
| `NEXT_PUBLIC_API_URL` | Backend API URL consumed by the frontend. Defaults to `http://127.0.0.1:8000`. |

Example:

```bash
NEXT_PUBLIC_API_URL=http://127.0.0.1:8000
```

## Usage

Start the backend and frontend in separate terminals:

```bash
# Terminal 1
cd backend
python manage.py runserver 127.0.0.1:8000

# Terminal 2
cd frontend
npm run dev
```

Useful local URLs:

- Frontend: `http://localhost:3000`
- Employee login: `http://localhost:3000/employee/login`
- Admin dashboard UI: `http://localhost:3000/admin1/dashboard`
- Django admin: `http://127.0.0.1:8000/admin/`
- Swagger API docs: `http://127.0.0.1:8000/api/docs/`
- Redoc API docs: `http://127.0.0.1:8000/api/redoc/`
- OpenAPI schema: `http://127.0.0.1:8000/api/schema/`

## API Overview

The backend exposes OpenAPI documentation at `/api/docs/`. Major API areas include:

| Area | Example routes |
| --- | --- |
| Auth | `/api/login/`, `/api/token/`, `/api/token/refresh/`, `/api/employee/login/` |

| Website content | `/api/services/`, `/api/gis-services/`, `/api/products/`, `/api/gallery/`, `/api/testimonials/` |
| Blog | `/api/blogs/`, `/api/blogs/<slug>/`, `/api/blogs/<slug>/comments/` |
| Vehicle permits | `/api/permits/vehicle-permits/`, `/api/permits/search-vehicle/` |

Prefer the generated Swagger UI for the complete request/response schema.

## Deployment

### Backend

1. Provision Python, PostgreSQL, and a production web server.
2. Set production environment variables.
3. Install dependencies:

   ```bash
   cd backend
   pip install -r requirements.txt
   ```

4. Run migrations and collect static assets:

   ```bash
   python manage.py migrate
   python manage.py collectstatic
   ```

5. Start the app with Gunicorn using the included examples:

   ```bash
   gunicorn myproject.wsgi:application -c gunicorn_config.py
   ```

6. Use the files in `backend/deployment/` as references for systemd and nginx.

For production settings, run Django with:

```bash
DJANGO_SETTINGS_MODULE=myproject.settings_production
```

### Frontend

The frontend includes both `vercel.json` and `netlify.toml`.

Generic production build:

```bash
cd frontend
npm install
npm run build
npm run start
```

Set `NEXT_PUBLIC_API_URL` to the deployed backend URL before building.

## Troubleshooting

| Issue | Resolution |
| --- | --- |
| Backend cannot connect to database | Confirm PostgreSQL is running and `DB_*` variables match your database. |
| CORS errors in browser | Add the frontend origin to `CORS_ALLOWED_ORIGINS` in Django settings. |
| Frontend points to wrong backend | Set `NEXT_PUBLIC_API_URL` before running or building the frontend. |
| Email/OTP not delivered | Verify SMTP variables and provider app-password settings. |
| Static/media files missing in production | Run `collectstatic`; confirm `STATIC_ROOT`, `MEDIA_ROOT`, and nginx/static hosting are configured. |
| Swagger docs fail to load | Ensure `drf-spectacular` is installed from `backend/requirements.txt`. |
